1.1 Lazy page allocation

1.1.1 前言

在 xv6 中，用戶向系統申請動態內存分配 (dynamic memory allocation, malloc) 時，系統會即時的給予 physical pages，但這樣的動作會有兩大問題：浪費記憶體 (分配了但從未被使用)、降低系統效能 (動態內存分配時需呼叫 syscall，而每次系統調用的同時都需要保存 user space 的上下文、切換 stack等操作)，為了解決這個問題，我們實作了一個常見的機制 lazy page allocation，這個機制在用戶申請動態內存時，不會被立即分配物理內存，而是在程序有需要時進行分配，以盡量減少不必要的 syscall，並提高效能。

1.2 three-level page table

1.2.1 前言

  • xv6 原始版本採用 two-level paging，2^10 * 2^10 = 2^20 個 kb 可定址範圍，再加上後面的 flag (2^12=4096 bytes)，一共是 2^20 * 2^12 = 2^32 bytes = 4GB，4GB 就是 x86 架構下可提供的最大定址。

  • 我們希望透過增加 level的方式，縮小原本的 page directory size，來達到加快搜索的速度：

將 xv6 第一層的 10 bits 切成 ( 2 + 8 ) bits，讓搜索方法變成 4 + 256 = 260，這樣的做法比原本的 xv6 ( 1024 ) 小，可以讓越後面的更容易找到。

🏃 實作過程(修改哪些檔案[含圖片])

3.1 Lazy page allocation

sbrk() 原始版本：修改 process size 以外也會增加其 physical memory size

sbrk() 第一次修改版本：當 process 被創建時不實際分配物理內存 (physical memory)，只修改 process struct 裡頭的 size 大小

sbrk() 第二次修改版本：based on 第一次的版本，額外對偏移值 (offset, n) 進行正負判斷，當偏移值為正時且符合記憶體空間限制時修改 process size，若為負則將多餘空間進行 deallocation

修改後的 sbrk() 因為系統只知道 process size 有變化，但是在內存中卻找不到分配的記憶體空間，因此會產生 page table fault，但原始的 xv6 沒有針對 page table fault 另外設計 trap case，所以需要實作一個新的 case 來解決，這邊定義 page table fault 的 trapno 為 T_PGFLT，當 kernel 發現了這個錯誤之後，會進行以下的動作:
- 尋找發生錯誤的地址空間 (儲存在 register cr2() 中)
- 使用 PRGOUNDOWN() 來向下取得 page 邊界
- 利用 mappages() 來將新申請的物理內存映射到錯誤的地址空間上，並更新 CPU 對應的狀態和 process struct
- 返回 user space，繼續執行當前 process

😎 結論

4.1 Lazy page allocation

  • 使用 lazy page allocation 後可減少大型動態記憶體分配時可能產生的 wasted memory。

4.2 three-level page table

  • 使用three-level page table 能讓我們較快地尋找資料。因為原本 two-level page 花 2^10 的時間去找，而 three-level page 則變成 2^2 + 2^8的時間。
  • 使用 ckalloc() 的連續空間能讓之後取地址資料更方便，但連續空間取得比較不易，且之後會造成 external Fragmentation 。
